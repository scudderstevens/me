Minimize systemic system inefficiencies with a single source of truth for web properties, service integrations and communication channels.

Challenge

Create a single source of truth across the broadest set of inter/intra systems; within a flexible architecture, framework and data models that can effectively manage growth and extend the solution's life cycle.

Create the initial service with a limited first use case that can efficiently evolve to meet the requirements of the long term serviceable area.

    Fix bugs, address user issues, and manage software/product changes.

Appropriate use of technology

I am painting with a broad brush so it is necessary to mention. Any solution should be appropriate for its intended use, or a solution should NOT be disproportionate to the problem. Yes, every i has a dot and t a cross, but creating a complex solution to serve up individual dots and cross marks would not be logical.
Another risk throughout the entire process was creating a solution for a nonexistent problem. Just because one can create something, anything, doesn’t mean it is worth the effort. There are strategies for answering the question of worthiness, but I will not be going into Cost Benefit Analysis or Return on Investment topics in this case study.

Why Verticals, axis X?

Following the corporate structure facilitated two design concerns. First, following the business structure as it grew, adding facilities, creating subsidiaries or consolidating assets, the necessary infrastructure would remain in sync. The Uni-Dir service and its communication channels would organically evolve with the business in parallel fashion.
Secondly, operationalizing corporate initiatives within a complex infrastructure would be directly correlated with expenses and returns. This simplified the budgeting and billing process, with tracking methods that were transparently integrated into a cost/benefit analysis process.

Technology Options

Robust by Design
Researching an effective technology stack

All software offerings present both positive and negative design concerns. The relationships of components that provide similar services creates a complex matrix of trade offs as a system grows. The research process focused on, Selecting the most appropriate options to provide a manageable balance of:

  • Business Requirements

  • Scalability

  • Security

  • Maintainability

  • Testability

Business Requirements

Content types and scope guide the selection of the technologies. To support adoption, usability and user friendly features were heavily weighted. From previous user research it was identified that archetypes spanned a broad set of skill levels. Therefore existing user journeys and capabilities needed to be respected; both could be barriers to the perceived value of the service.

Scalability

To support increased workloads certain resource system requests were identified as potential bottlenecks. To address this, content delivery networks (CDNs) were identified as a requirement to create additional buffers to high volume or large file system calls. Since future service needs were not fully known, the initial focus was to keep the majority of the service requests synchronous in nature.

Security

The long term goals required supporting secured traffic* but the use cases were only partially defined. This focused security concerns to back-end dashboard access, and providing user rights/privileges that honored existing user journeys. This resulted in workflows that were limited to Archetypes and user centric processes.

Maintainability

As the system evolves, regular updates, bug fixes, and enhancements would be required. Therefore, keeping these necessities efficient would significantly reduce the time, effort, and cost of these updates. To improve maintainability modular data models and their relationships would be implemented and a system strategy clearly documented.

Testability

To ensure that the system's individual components and the system as a whole behaved as expected, a suitable continuous development tooling and process would be required. Special care was taken, focusing on development and maintenance strategies, decoupling components and data structures to maximize long term flexibility.

*beyond Hypertext Transfer Protocol Secure (HTTPS) protocol, i.e. intra-network communications.

Bonus Points

Not mentioned within the solution described, the MVP provides a solution for two problems. The product design documented requires an API to access the tax content and analytical data necessary for the solution to function. This API and the logic behind the functionality has not yet been created or a suitable use case identified. The completed MVP (product) would initiate the boot-strapping of a Content as a Service (CaaS) platform, opening up the possibility of future product variations.

Takeaways

The tax research tool proposed at the conclusion of Phase Two, proven to be worthy of progressing to the MVP Phase, would be developed using existing Vertex content. A limited subset of that content would provide Sales Tax both state, local and provide all appropriate citations. The tax research tool supplements the tax content available in other Vertex solutions. The initial iteration of the tool would maintain a narrowly focused (minimum viable) use case and scope. Future iterations of the tool would incorporate the remaining tax content and eventually expand the content library into new areas of research.
